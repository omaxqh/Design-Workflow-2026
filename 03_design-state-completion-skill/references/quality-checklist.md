# 质量检查清单

本文件只做交付前验收，不复述 `execution-pipeline.md` 的完整流程。阶段执行规则以 `execution-pipeline.md` 为准；高频失败模式以 `failure-patterns.md` 为准。

## 使用方式

在阶段 6 截图 QA 和最终交付前逐项检查：

- 命中问题：先修复，再交付。
- 无法确认：进入“需要人工确认的问题”。
- 平台基础状态、骨架态或复盘任务只检查相关条目，不强行套用全部清单。

## 阶段 1-3：推演与矩阵

- 是否使用 `decision-framework.md` 完成需求对象定位。
- 是否判断了对象层级：页面级 / 模块级 / 子结构级 / 字段级。
- 是否识别目标对象内部变量：类型、数量、互斥、优先级、长度、可用性。
- 是否列出稳定字段，避免误改非目标字段。
- 是否区分业务状态和平台基础异常态。
- 是否覆盖参与前、参与中、结果后，以及资格、时间、权益、库存、交易等关键变量。
- 是否过滤不适用组合，并说明原因。
- 状态矩阵是否包含主页面已完成状态和待补齐状态，而不是只列缺失状态。
- 矩阵横纵轴是否来自业务变量组合，而不是画面文案差异。
- 用户要求补子结构全情况时，矩阵主轴是否对齐该子结构内部变量。
- 每个状态是否有触发条件、用户含义、页面变化、主 CTA、组件生成方式、是否画稿和假设说明。

## 阶段 2-5：组件与画稿

- 是否读取主页面已有 component / component set、实例、main component 和 variant properties。
- 是否明确每个状态的生成方式：原组件覆盖 / 补充 variant / 结构扩展 / 仅进矩阵。
- 是否判断补充 variant 应进入哪个原父 component set。
- 是否避免用手绘近似稿替代已有组件实例。
- 颜色、按钮、文字链、分隔线、圆角、间距是否来自原组件或原页面。
- 页面级状态是否用完整页面呈现，而不是缩略线框。
- 页面组合态是否优先复制主页面或主模块实例，优先切换 variant，再做必要 override。
- 必要结构变体是否已标注，且没有破坏原视觉语言。
- 状态标注是否优先使用纯文本，避免无业务意义的装饰画板。
- 写入 Figma 时是否创建在原文件新区域，且没有覆盖原主页面。

## 内容边界与布局

- 数量型内容边界是否做了合理布局重排，而不是机械删除、拉宽或缩放。
- 单卡 / 双卡 / 多卡状态是否分别检查信息结构、图片比例、文案行数、模块高度和下方模块位置。
- 减少数量时，是否说明保留对象及依据，而不是默认取前 N 个。
- 是否区分排布能力和展示能力，例如可 wrap 但固定高度裁切时，没有机械增高容器。
- 空状态是否判断删除模块还是保留占位；无占位要求时是否删除区域并让下方内容顺延。
- 超长文案是否真实触发换行、截断或溢出边界，而不是被缩短成普通单行。
- 标题与标签共用 wrap 容器时，是否继承自然换行逻辑。
- 如果手动模拟换行或定位，是否测量文字宽度并截图确认无空洞、孤行、叠字或裁切。

## 阶段 6：截图 QA

- 是否使用 `failure-patterns.md` 主动检查高频失败模式。
- 是否确认没有矩阵主轴错位、非法组合误生成、空状态留白。
- 是否确认组件复用没有被手绘替代。
- 是否确认 variant 切换和文案 override 真实生效。
- 是否检查并使用可加载中文字体，截图中没有中文缺字、空白或裁切。
- 截图目标是否为真实可渲染 Frame / Instance / Component，避免截到空白 Section。
- 是否抽样复查主态与补充态关键几何值：容器宽高、内边距、标题到内容间距、底部 padding、关键标签 / 徽标实例复用。
- 是否区分稳定字段和状态字段。
- 下方模块是否继承原 gap / padding 顺延，没有用固定 y 值造成状态间距异常。
- 固定底 bar、悬浮入口、安全区是否没有被误当作页面流内容移动。
- auto layout 子节点顺序是否正确，例如权益 / 内容说明在活动行之前。
- 页面级状态是否完整可读。

## 平台基础状态专项

仅当用户明确要求 loading、骨架屏、空态等平台基础状态时检查：

- 是否读取平台规范源文件或设计系统源节点，而不只是导入单个示例组件。
- 检索平台规范时是否拆分中文核心词和类型词。
- 是否只生成用户需要的基础状态，没有顺手扩写无关异常态。
- 骨架态是否保留原模块容器、内边距、标题区、列表区、列间距和元素边界。
- 规范色在当前背景不适配时，是否做了有说明的语境映射。
- 如果规范包含动画说明，静态稿是否标注透明度范围、循环速度和默认呈现状态。

## 阶段 7：复盘沉淀

- 是否对比自动稿与用户反馈或调试期对照稿的输出结构。
- 是否识别页面级状态、模块级矩阵、结构变体的差异。
- 是否把差异归因到场景层级、业务变量、字段联动、组件复用、Figma 执行或画布组织问题。
- 是否区分通用规则和当前样本特有规则。
- 是否提出建议沉淀项和目标文件，并等待用户确认。
- 用户确认后，通用规则是否写入对应 reference；调试样本特有结论如需保留，是否只写入隐藏内部调试记录而非正式流程入口。

## 最终回复

- 是否包含已补全状态数量。
- 是否包含状态矩阵摘要。
- 是否包含需要人工确认的问题。
- 人工确认问题是否只列真正影响业务判断的问题。
- 人工确认问题是否优先写成选择题，并说明每个选项对状态矩阵或画稿的影响。
