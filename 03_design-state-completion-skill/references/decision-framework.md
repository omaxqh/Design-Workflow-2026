# 状态补全判断框架

本框架用于替代“靠案例记忆”的推导方式。每次进入状态矩阵前，必须先完成对象、变量、展示策略和稳定字段判断。

## 1. 需求对象定位

先判断用户真正要求补齐的对象层级：

- 页面级：页面结构、导航、列表数量、主流程变化。
- 模块级：卡片、权益模块、房型 rate、订单模块等整体状态。
- 子结构级：标签组、权益区、价格区、CTA 区、标题区、说明区。
- 字段级：名称、价格、地址、规则文案、标签文案等内容边界。

如果用户明确说“某个子结构的全情况”，矩阵主轴必须来自该子结构内部变量，不要上升成笼统业务概括。

## 2. 变量识别

对目标对象列出可组合变量：

- 类型：有哪些子元素或业务字段。
- 数量：0 / 1 / 多个，是否有上限。
- 互斥：哪些类型不能同时出现。
- 优先级：空间不足时谁先展示，谁后置或裁切。
- 长度：文案、价格、地址、标签是否存在较长/超长状态。
- 可用性：可用、不可用、部分可用、已使用、未满足资格等。

变量必须来自目标对象本身。不要用抽象命名代替真实变量，例如“高价值标签”只能作为解释，不应替代“二官权益合标 / 运营坑标 / 普通服务标签”成为矩阵主轴。

## 3. 合法组合过滤

先列组合，再过滤不成立组合：

- 根据数量上限过滤，例如某类标签最多 1 个。
- 根据互斥关系过滤，例如同一价格表达不能同时作为主价。
- 根据业务意义过滤，例如用户不可能同时未参加且已完成。
- 根据低价值判断降级，例如只改变无关文案且不影响用户决策。

被过滤的组合应进入“不适用组合列表”，说明过滤原因。不要把非法组合生成成画稿后再修。

## 4. 展示策略判断

对每个合法组合判断展示策略：

- 增高：内容需要完整展示，且原模块允许页面流顺延。
- 裁切：容器固定高度，超出内容可排布但不展示。
- 折叠：超过阈值后用“更多 / 展开 / X项”等入口承载。
- 删除：无内容且没有占位要求时移除模块。
- 占位：规范要求布局稳定或异步加载时保留空间。
- 顺延：模块高度变化后，下方内容按原 auto layout 或原 gap/padding 顺延。

必须区分“排布能力”和“展示能力”。例如容器可以 wrap 只说明它可排到下一行；如果高度固定且 clipsContent=true，就不代表下一行会被展示。

## 5. 稳定字段锁定

生成前必须列出稳定字段，避免状态补齐误改：

- 业务稳定信息：酒店名、房型名、店铺名、活动名、权益名、基础价格结构。
- 视觉稳定组件：按钮样式、图标、徽标、标签样式、圆角、分隔线。
- 布局稳定关系：原 gap、padding、下方模块位置、固定底 bar、安全区。

只有目标对象相关字段可以变化。其他字段如需变化，必须说明它与该状态的联动关系。

## 6. V1 交付前判断

V1 不是草稿。交付前必须完成：

- 语义 QA：矩阵主轴对齐对象，组合合法，状态名称来自真实变量。
- 组件 QA：使用原组件 / 原实例 / variant，variant 切换真实有效。
- 内容 QA：文案 override 成功，没有缺字、叠字、裁切异常。
- 布局 QA：间距、字号、行内关系、下方顺延符合原稿。
