# Figma 生成规则

## 工具规则

- 写入 Figma 前必须使用 `figma-use` skill。
- 只有用户明确要求写回 Figma 或生成画稿时，才写入用户提供的原 Figma 文件。
- 写入前必须确认目标 file / node、补全区域名称和写入范围。
- 不覆盖原主页面。
- 优先复制主页面结构后改造，不从空白重画。
- 如果存在设计系统组件，优先复用。
- 如果缺少组件，可基于主页面已有模块风格生成。
- 如果原稿中存在组件实例和 variants，优先复制组件实例并切换 variants，再做必要文案调整。
- 组件实例优先级高于视觉近似。即使手动画稿里的某个状态看起来像普通 frame，也要先判断主页面或组件区是否已有可复用 component / component set。
- 不允许为了“看起来相似”手绘替代已有组件。颜色、按钮、文字链、分隔线、圆角、间距应来自原组件。
- 当需求是把同一业务表达从一种组件类型补到另一种组件类型时，应先寻找目标组件自己的承载字段和子实例，不要直接复制来源组件的结构。例如房型 rate 的 88VIP 权益表达补到普通 rate 时，必须基于普通 rate 原组件实例改早餐、退改政策、运营坑标等字段。

## 优先级与冲突消解

当规则之间看起来冲突时，按以下顺序判断：

1. 用户明确要求优先
   - 用户要求完整页面，就生成完整页面。
   - 用户要求只输出矩阵或只补组件，就在对应阶段停止。
2. 业务交付层级优先
   - 状态改变页面结构、模块数量、导航、列表或用户路径时，输出页面级状态。
   - 状态只发生在一个模块内部时，优先输出模块矩阵，必要时补页面组合态。
3. 组件复用优先
   - 有原组件或原实例时，先复制和改造原结构。
   - 允许结构重排，但不允许用手绘近似替代已有组件视觉语言。
4. 布局合理性优先于机械保持
   - 内容数量、密度或布局方向变化时，可以改结构、压缩高度或切换卡片方向。
   - 改结构时必须保留原色彩、圆角、字体层级、图文风格和关键装饰关系。
5. 固定区域判断优先
   - 页面流内容可随模块压缩上移。
   - 固定底 bar、悬浮入口和安全区通常保持位置，除非原页面结构表明它们属于普通页面流。

## 命名规则

所有节点命名使用中文。

顶层区域：

`业务状态补全 / {页面名称}`

推荐结构：

- `00 状态矩阵`
- `01 页面级场景`
- `02 核心模块全情况`
- `03 结构变体`
- `04 子组件变体`

不要默认创建 `99 待确认问题`。需要人工确认的问题只在最终回复中输出；只有用户明确要求沉淀到 Figma 时，才写入 Figma。

单个画稿命名：

- `{页面名称} / 未参加`
- `{页面名称} / 挑战中`
- `{页面名称} / 挑战成功`
- `{页面名称} / 不可报名`
- `{页面名称} / 会员等级不足`
- `{页面名称} / 权益已使用`
- `{页面名称} / 价格变动`
- `{页面名称} / 日期无房`

状态标签命名：

`状态说明 / {状态名称}`

## 矩阵布局

如果是模块全情况，优先用二维矩阵摆放。

示例：

- 横轴：参与状态
- 纵轴：任务类型

挑战卡示例：

- 横轴：未参加 / 挑战中 / 挑战成功 / 挑战失败结束 / 不可报名
- 纵轴：单次任务 / 单进度任务 / 多阶梯任务

酒店会员卡示例：

- 组件变体区：按子组件分组，分别补标题区、权益展示元素、权益区详情、进度条、提示气泡、活动行。
- 页面模块状态区：按状态纵向列举未开卡、激活账号、注册中、会员失效、无房晚积分、有房晚积分、挑战任务中、即将升级、3 权益、2 权益、1 权益、无活动。
- 权益布局矩阵：横轴可为权益数量 4/3/2/1，纵轴可为可享 / 不可享 / 部分不可享。

## 画稿规则

- 保持原页面视觉语言。
- 允许改动页面结构。
- 改动页面结构时必须标记为 `结构变体`。
- 业务状态变化要体现在 CTA、文案、状态标签、模块显隐、利益点表达或流程反馈中。
- 业务稳定元素不要随状态变化，包括挑战名称、任务标题、奖品名称、会员等级名称。
- 页面级状态必须用完整页面呈现，保留头图、导航、背景、核心模块、奖品中心和活动规则，不要只画缩略线框。
- 生成前必须检查可加载中文字体。若 `PingFang SC` 不可加载，优先使用 `Noto Sans SC` 或当前文件中可加载的中文字体。
- 输出后必须截图检查中文文本是否缺字、裁切或显示为空。
- 当用户要求“直接生成为子组件”时，必须创建或补齐 Figma component / component set，而不是只画普通 frame。
- 补充变体必须和原变体放在同一个父 component set 内；不要把“原组件”和“补充变体”分散成两个无关组件区。
- 如果需要区分来源，可在变体标题中标注 `来源=原组件`、`来源=补充`，但父组件仍保持同一组。
- 子组件补齐后，必须再生成页面内组合态，验证子组件在真实页面模块中的承接效果。
- 对复合模块，应先拆子组件，再列页面组合态。不要把所有状态都塞进一个大组件矩阵。
- 页面组合态中的状态标注优先用纯文本，不要为了标注状态额外创建装饰性画板或标签组件，除非原设计规范已有状态标签组件。
- 当复制原页面模块后，如果父容器是 auto layout，必须检查子节点顺序。权益 / 内容说明应在活动行之前，活动行应在内容之后；不能只改 y 坐标。
- 对 rate、卡片、列表项等已有组件实例，状态强化应优先修改可变子实例：利益点标签、权益标、服务标签、退改政策、早餐、价格利益点、CTA 文案；稳定字段如酒店名、店铺名、基础价格结构和按钮样式不要随状态误改。

### 页面组合态的页面流联动

- 当用户提供完整页面节点时，即使主能力是内容边界布局，也必须判断是否附加 `页面组合态生成` 作为辅助能力。
- 写入完整页面组合态前，必须先建立“后续页面流对象清单”：目标模块之后的列表项、依附在列表项上的标签 / 浮层 / 价格说明、固定底 bar、悬浮入口和安全区。
- 模块高度、模块显隐或内容数量变化后，若下方对象属于页面流，应按 `新 y = 上一对象 y + 上一对象实际高度 + 原 gap` 重新排布，保持列表连续；不得只移动部分列表项。
- 依附对象必须跟随所属对象移动，例如酒店卡片上的权益价值标签、价格浮层、角标、徽标，不得只移动卡片本体导致标签脱离。
- 若用户要求“页面其他内容不动”，但模块高度收缩会产生明显大空白，应暂停确认：保留原位并允许空白、按页面流顺延，或只生成模块态不生成完整页面组合态。
- 完整页面组合态 QA 必须同时检查模块局部截图和整页截图；模块局部合格不代表页面流承接合格。

### 状态矩阵对象对齐

- 当用户明确要求补某个子结构的全情况，如标签组、权益区、价格区、CTA 区，状态矩阵主轴必须来自该子结构内部变量，不要上升成笼统业务概括。
- 对复合子结构，必须先识别内部元素类型、数量上限、互斥关系和优先级，再生成合法组合矩阵。不成立的组合应在矩阵阶段过滤，不要生成后再修。
- 必须区分排布能力和展示能力。例如子结构可以 `wrap`，但如果容器高度固定且裁切开启，说明它可以排到下一行但不一定展示下一行，不能机械增高容器。
- 空状态必须判断是删除模块还是保留占位。若某个子结构没有内容且没有规范要求稳定占位，应删除该区域并让下方内容按原 auto layout 上移，不要保留无意义空白。

### V1 原稿延续性 QA

- V1 也必须达到可验收的 UI 延续性。不要把布局、间距、字号、元素复用、组件实例复用等基础问题留到 V2/V3 修正。
- 生成 V1 前必须至少抽样复查一个主态与一个补充态的关键几何值：容器宽高、内边距、标题区高度、标题到内容间距、底部留白、关键标签/徽标是否仍为原实例。
- V1 交付前必须完成一次原稿延续性 QA：组件复用、字号、颜色、圆角、间距、行内关系、下方模块顺延都要通过。
- V1 交付前还必须完成一次语义 QA：状态名称是否来自真实对象变量，组合是否业务成立，组件 variant 切换是否真实可用，文案 override 是否成功。
- 如果原稿下方内容跟随标题区、权益区、列表区等上方内容自然顺延，必须继承原 gap 和 padding 关系。不得用固定 y 值批量移动下方模块；应按 `内容区 y = 上方区域 y + 上方区域实际高度 + 原 gap` 校验。

### 超长状态补齐

- “超长状态补齐”包括名称、标题、地址、规则、价格、标签组等内容边界导致的较长、超长、截断、换行和溢出状态。
- 对内容边界状态，如标题超长、标签增多、价格位数变长、地址/规则文案过长，应优先保留原模块 auto layout、原组件实例和原子节点顺序，只调整触发边界的文本框宽高、最大行数、截断和必要的容器高度。不要手工重画标签、徽标、按钮、图标或下方稳定模块。
- 内容边界状态首先要保留状态强度：长就是长，超长就是超长。不能为了继承局部布局规则，把超长样例缩短成普通单行。
- 其次才是继承原稿布局逻辑：标签能跟随就跟随，不能机械下移或统一独占整行。
- 标题 + 标签共用横向 wrap 容器时，必须继承原稿的机动换行逻辑：标题未占满当前行时，后续标签应继续贴在同一行；只有剩余宽度不足时才换行。不要把所有“较长标题”机械处理为标题独占整行或固定两行。
- 如果原稿依赖 wrap，必须先识别自然换行规则，再决定是否需要手动模拟；不要在未理解行内流动关系前直接拆行或固定定位。
- 一旦手动模拟文本换行、拆分文本行或改为绝对定位，必须先测量当前字体下的真实文字宽度，再截图确认没有右侧无效空白、行内空洞、单个徽标孤立成行、文本内部换行叠字或裁切。只要出现这些基础问题，应回退为更贴近原 auto layout 的实现或重新计算行宽。
- 对数量型内容边界，不要只删除多余卡片或等比例拉伸剩余卡片。必须根据状态重排布局：
  - 多列到双列：可保持原卡片结构，但要重新分配宽度、文案行数、内容区高度和模块高度。
  - 多列到单卡：优先判断是否应切换为横向图文、重点卡或摘要卡，而不是继续使用上图下文的大卡。
  - 内容减少后，下方模块应按页面流重新排布；但固定底 bar、悬浮入口和安全区通常保持原位置，不跟随内容压缩上移。
  - 保留项不应默认取前 N 个，应根据业务优先级、主页面重点和用户决策价值选择。
- 如果使用 Figma Section 组织画布，必须确认截图目标包含真实子节点；否则应使用 Frame 作为可渲染承载容器。截图 QA 发现空白 Section 时，要删除或修正本次生成的无效节点后再交付。
- 不要为了凑数量生成低价值状态。
- 不要把平台基础异常态当作主要产出。
- 对不确定业务规则使用假设标注。
- 不生成业务上不成立的组合。

## 平台基础状态与规范读取

当用户明确要求生成 loading、骨架屏、缺省、空态等平台基础状态时，可以将该基础状态作为本次主要产出，但必须收敛范围，不扩写无关异常态。

平台规范读取规则：

- 不能只在当前业务文件中搜索规范组件，还要检查已订阅设计系统和用户提供的规范源文件。
- 设计系统搜索关键词必须拆分检索。不要只用 `loading skeleton 骨架屏 加载 占位` 这类混合查询；应分别检索 `骨架`、`骨架型`、`加载`、`加载态`、`占位`、`通用型`、`下拉型`、`图片型` 等核心词。
- 如果远程组件的 `description` 和 `documentationLinks` 为空，不能认为已读取完整规范。应继续寻找源规范页或请求用户提供规范源文件链接。
- 读取源规范页时，应抽取页面级说明文字、同组资产类型、关键色值、圆角、动画、透明度、速度和适用场景，而不是只导入某个组件实例。
- 规范资产引用应作为证明材料进入补全区域，但核心画稿仍应适配当前业务模块版式。

骨架型适配规则：

- 白色背景规范色不能机械套用到深色背景。若规范色明显不适配当前模块背景，应保留结构、圆角和动效规则，对颜色做语境映射，并在画稿说明中标注映射原因。
- 骨架态应保留原模块的结构节奏：容器尺寸、内边距、标题区宽高、列表行宽高、列间距、icon 容器位置和文案框高度应优先从原稿读取。
- 骨架块高度应对齐被替代元素的可视区域。文本骨架不应只按文字视觉高度估算，而应检查原 text node 的 frame 高度。
- 如果原矩形已有明确圆角，应继承原矩形圆角；否则使用平台规范的一般圆角。
- 静态 Figma 稿应呈现默认透明度状态，并可补充 40% 对照样例说明呼吸动画。

## 状态矩阵规则

状态矩阵必须先于画稿生成。

矩阵字段：

- 状态名称
- 状态类型
- 触发条件
- 用户含义
- 页面变化
- 主 CTA
- 是否画稿
- 优先级
- 假设说明

矩阵应放在补全区域第一组。
