# 复盘沉淀规则

本文件只定义复盘沉淀方法：如何对自动稿问题、用户反馈，以及调试期对照稿差异做归因，并决定规则写入位置。正式生成、盲测、内部调试资产和标准答案的边界以 `execution-pipeline.md` 为准。

## 使用时机

当用户提供以下任一材料时，必须执行复盘沉淀：

- 自动生成的精修版 Figma 链接
- 用户对自动稿提出的明确问题截图或文字反馈
- 调试期设计师手动完成的全情况 Figma 链接
- 调试期用户说明“这个人工稿是我想达到的效果”

如果只有主页面和用户反馈，没有调试期对照稿，也必须做问题归因和规则沉淀；不要等待标准答案。

## 复盘输入

最小输入：

- 自动稿 Figma 节点
- 需求背景
- 用户指出的问题；调试期可补充对照稿 Figma 节点

建议补充：

- 用户认为最重要的 3-5 个状态
- 调试期用户不希望学习的人工稿部分
- 自动稿被指出的问题截图
- 业务规则补充说明

## 复盘输出

每次复盘必须输出三类结论：

1. 差异摘要
   - 自动稿缺了什么
   - 自动稿多了什么
   - 自动稿表达错了什么
   - 自动稿结构是否偏离用户反馈或调试期对照稿
2. 可复用规则
   - 哪些规则可以迁移到其他业务
   - 哪些只适用于当前业务域
   - 哪些是 Figma 生成约束
3. skill 更新项
   - 应更新哪个参考文件
   - 新增规则的触发条件
   - 新增规则的验收方式
   - 对应的推演能力分类

## 对比维度

### 1. 输出结构

检查自动稿是否覆盖了用户反馈或调试期对照稿指出的主要结构层级：

- 页面级状态
- 模块级矩阵
- 结构变体
- 业务说明或状态矩阵

规则：

- 如果对照信息同时包含页面级状态和模块级矩阵，自动稿也必须同时保留。
- 页面级状态必须用完整页面呈现，不能用缩略线框替代。
- 模块矩阵应独立成区，不应混在页面级状态里导致难以扫描。

### 2. 状态覆盖

检查自动稿是否遗漏业务关键状态。

必须关注：

- 参与前
- 参与中
- 结果后
- 资格不满足
- 时间窗口变化
- 权益发放或使用结果
- 单对象 / 多对象
- 单任务 / 多任务

规则：

- 不要按字面状态穷举，要识别背后的业务变量。
- 复盘中反复出现的状态维度，应升级为通用矩阵维度。
- 自动稿中只有文案差异、没有业务意义的状态，应降级为矩阵说明或删除。

### 3. 状态字段联动

检查自动稿是否只改了状态标签，而没有改卡面元素。

必须检查：

- 时间文案
- 状态标签
- 进度值
- 辅助说明
- 奖励结果
- 行动点
- 主按钮文案
- 主按钮显隐
- 锁定 / 置灰状态

规则：

- 状态变化必须带动相关字段联动。
- 可报名态不应展示挑战剩余时间，应展示报名时间或报名窗口。
- 进行中态应展示剩余时间、当前进度和下一步行动。
- 成功、失败、不可报名等无可用操作状态，通常不展示主按钮。

### 4. 稳定字段保护

检查自动稿是否误改了不该随状态变化的业务字段。

通常稳定的字段：

- 页面标题
- 商家名称
- 活动名称
- 挑战名称
- 任务标题
- 奖品名称
- 会员等级名称
- 任务目标名称

通常可变的字段：

- 时间文案
- 参与状态
- 进度数值
- 辅助解释
- 操作文案
- 结果文案
- 权益发放状态

规则：

- 同一个任务在不同参与状态下，任务标题应保持一致。
- 不要根据当前文本内容判断字段类型；当前文本可能已经被错误写入。
- 字段类型判断优先级：图层名 > 组件层级 > variant 属性 > 业务语义 > 当前文本。

### 5. 组件复用程度

检查自动稿是否遵循 `figma-generation-rules.md` 中的组件复用规则，并记录偏离点。

复盘时只归因，不在本文件复述具体生成方法。

### 6. 视觉与布局质量

检查自动稿是否便于设计师扫描，并对照用户反馈或调试期对照稿判断差异。

必须检查：

- 矩阵横纵轴是否清晰
- 页面级状态是否成组
- 状态标签是否可读
- 卡片间距是否稳定
- 文字是否裁切或丢失
- 是否出现空白文本
- 是否有多余说明污染画布

规则：

- 如果对照信息采用紧凑矩阵，自动稿也应优先保持紧凑矩阵。
- 状态说明可以放在矩阵标题或底部短注释中，不要压到核心画稿上。
- 需要人工确认的问题默认只在最终回复中输出，不写入 Figma。

### 7. 业务假设与确认问题

检查自动稿是否把不确定规则当成确定规则。

规则：

- 不确定但影响画稿的规则，必须进入“需要人工确认的问题”。
- 确认问题优先写成选择题。
- 只询问会改变状态矩阵、CTA、页面结构或字段展示的问题。
- 纯文案偏好、视觉微调不应默认作为人工确认问题。

## 差异归因方法

每个差异都要归入一个原因，避免只写现象。

先按 `capability-routing.md` 的能力分类判断差异属于哪一种推演能力失败，再判断具体原因。本文件只记录复盘归因方法，不重复维护能力分类清单。

常见归因：

- 场景层级遗漏：只做模块矩阵，漏了页面级状态。
- 业务变量遗漏：没有识别单任务 / 多任务、单对象 / 多对象等变量。
- 状态联动不足：只改状态标签，没有同步修改文案、按钮、进度。
- 字段稳定性误判：把任务标题、挑战名称等稳定字段改成状态文案。
- 组件复用不足：没有复制原组件实例或没有正确切换 variant。
- Figma 执行问题：字体不可用、文本裁切、隐藏层、节点定位错误。
- 画布组织问题：分区过重、矩阵不清晰、说明信息干扰画稿。

## 规则沉淀位置

复盘后按规则类型更新对应文件：

- `scenario-decomposition.md`
  - 场景层级
  - 业务变量
  - 页面级 / 模块级拆分
  - 不适用组合判断
- `business-state-taxonomy.md`
  - 新增业务状态类型
  - 状态命名
  - 触发条件归类
- `fliggy-domain-patterns.md`
  - 飞猪酒店域、会员域、活动权益域的业务模式
  - 域内常见状态组合
- `figma-generation-rules.md`
  - Figma 写入结构
  - 组件复用
  - 字体处理
  - 画布布局
- `quality-checklist.md`
  - 交付前必须检查的问题
  - 截图 QA 项
  - 失败高发点
- `capability-routing.md`
  - 新增或调整推演能力分类
  - 能力命中条件
  - 分类下的检查重点和禁止误用规则
- 隐藏内部调试记录，仅限调试 / 回归评测需要
  - 具体样本背景
  - 自动稿版本问题
  - 最终验收结论
  - 该样本特有规则

不要把可迁移经验堆进内部样本文件。可以迁移到其他需求的内容应进入正式规则文件；内部样本不得被正式生成流程引用。
