# 执行流水线

本文件定义 `design-state-completion` 的固定执行阶段。每次使用 skill 时必须先判定流程模式，再完成能力路由，并按阶段推进，不应跳过中间产物直接写 Figma。

## 流程模式判定

进入阶段 0 前先判定模式：

- 盲测 / 泛化测试：用户提供新需求，希望验证 skill 能否自行推导。只读取主页面、需求背景、组件结构和能力规则；不读取手动全情况、标准答案或内部调试资产。
- 正式生成：用户明确要求写回 Figma 或生成画稿时，按阶段 0 能力路由和阶段 1-7 执行流程推进，并写入原 Figma 文件的新区域。
- 调试复盘：用户提供手动全情况、指出自动稿问题，或要求总结差异。允许读取手动全情况，只做差异归因和规则沉淀，不进入正式生成链路。

如果用户同时提供主页面和手动全情况，但表达的是“测试泛化能力”或“先跑小测试”，按盲测 / 泛化测试处理，手动全情况暂不读取。

## 阶段 0：推演能力路由

目标：先判断当前需求主要调用哪一种补全能力，再决定执行哪些规则检查清单。正式生成时只依赖主页面、业务来源、组件结构和能力规则，避免按页面表面相似度套用旧样本。

本阶段必须使用 `capability-routing.md`。能力分类和检查重点以该文件为准；这里只规定最多选择 1 个主能力和 1 个辅助能力。

必须输出：

- 主能力
- 辅助能力
- 判断依据
- 需要执行的规则检查清单
- 不适用能力及原因

内部调试资产规则：

- 正式生成、盲测和泛化测试均不得读取、引用或展示内部调试资产。
- 内部调试资产只用于 skill 维护、调试期复盘和离线回归评测。
- 调试期标准答案只能用于差异归因和规则抽象，不得复制节点作为生成稿。

退出条件：

- 已明确主能力和辅助能力。
- 已明确本次需要执行的能力检查清单。
- 已确认不依赖内部调试资产或标准答案进入正式生成。

不得进入阶段 1 的情况：

- 未完成能力路由。
- 选择多个主能力。
- 依赖内部调试资产或标准答案才能继续推演。

## 阶段 1：业务状态推演

目标：理解页面业务，而不是先列 UI 状态。

本阶段必须使用 `decision-framework.md`。先完成需求对象定位，再识别变量，不要直接平铺状态名称。

输入：

- 主页面 Figma 链接
- 需求背景
- 业务域
- 用户明确要求的状态

正式使用时默认没有手动全情况稿或标准答案。阶段 1 必须基于主页面、需求背景、组件结构和阶段 0 命中的能力规则独立产出推演结果。

如果当前任务是调试期复盘，用户主动提供的手动全情况稿只能在阶段 7 读取，用于差异归因和规则抽象。

必须分析：

- 核心业务对象
- 需求对象层级：页面级 / 模块级 / 子结构级 / 字段级
- 用户任务
- 页面级场景
- 模块级状态
- 子结构或字段级变量
- 业务变量组合
- 内容边界状态
- 不适用组合
- 稳定字段

输出：

- 业务对象清单
- 需求对象层级判断
- 页面级场景清单
- 模块级状态变量清单
- 子结构 / 字段变量清单
- 候选状态池
- 不适用组合初判
- 稳定字段清单

退出条件：

- 已说明状态发生在哪一层：页面级、模块级、子组件级或内容边界。
- 已区分业务状态和平台基础异常态。
- 已标注不确定业务规则。

不得进入下一阶段的情况：

- 只列了状态名称，没有触发条件。
- 没有判断页面级和模块级边界。
- 把通用加载失败、网络异常当成主要产出。

## 阶段 2：组件复用预检

目标：写稿前先确认 Figma 中已有组件覆盖范围，避免手绘近似稿。

输入：

- 主页面节点
- 组件区或本地组件库
- 页面内核心模块实例

必须检查：

- 主页面是否已有 component / component set
- 核心模块是否由实例组合而成
- 每个实例的 main component 和 variant properties
- 哪些状态可以通过切换 variant 覆盖
- 哪些状态需要新增 variant
- 哪些状态只能通过结构扩展实现
- 原模块父容器是否使用 auto layout

输出：

- 组件覆盖表
- 原组件可覆盖状态列表
- 需补充 variant 列表
- 结构扩展列表
- 组件复用风险

退出条件：

- 已明确每个核心状态的生成方式：原组件覆盖 / 补充 variant / 结构扩展 / 仅进矩阵。
- 已明确补充 variant 应进入哪个原父 component set。
- 已确认不得手绘替代已有组件。

不得进入 Figma 写稿的情况：

- 未读取组件实例和 component set。
- 不知道哪些状态能由原 variant 覆盖。
- 未判断补充 variant 的父组件归属。

## 阶段 3：状态矩阵

目标：把业务状态转成可执行矩阵，并过滤低价值或不成立组合。

矩阵必须基于 `decision-framework.md` 的变量组合推导。若用户明确要求补某个子结构的全情况，矩阵主轴必须来自该子结构内部变量。

输入：

- 阶段 1 的候选状态池
- 阶段 2 的组件覆盖表

矩阵字段：

- 状态名称
- 状态类型
- 触发条件
- 用户含义
- 页面变化
- 主 CTA
- 组件生成方式
- 是否画稿
- 优先级
- 假设说明

输出：

- 状态矩阵
- 不适用组合列表
- 只进矩阵不画稿列表
- 必须画稿列表
- 矩阵主轴说明
- 非法组合过滤原因

退出条件：

- 每个状态都有触发条件和用户含义。
- 每个状态都标明生成方式。
- 不适用组合已过滤。
- 低价值状态已降级或删除。
- 矩阵主轴已对齐需求对象。

## 阶段 4：子组件变体

目标：复合模块先补齐子组件系统，再组合页面状态。

适用条件：

- 用户要求“直接生成为子组件”
- 主模块由多个子组件组成
- 状态差异主要发生在标题区、权益区、按钮区、活动行等子模块

规则：

- 按 `figma-generation-rules.md` 的组件变体规则执行。
- 必须说明补充 variant 的父组件归属、生成方式和变体命名。

输出：

- 子组件变体区
- 原变体列表
- 补充变体列表
- 变体命名说明

退出条件：

- 补充 variant 已放入正确父组件。
- 组件样式来自原组件。
- 没有用手绘近似稿替代原组件。

## 阶段 5：页面组合态

目标：用真实页面或真实模块实例组合业务状态，验证子组件在页面中的承接效果。

规则：

- 按 `figma-generation-rules.md` 的页面组合态规则执行。
- 页面级状态必须用完整页面呈现；模块状态必须放回真实页面或真实模块中验证承接效果。

输出：

- 页面级场景
- 模块组合态
- 结构变体

退出条件：

- 每个必须画稿状态都有画稿。
- 页面组合态中的关键模块来自原实例或原组件扩展。
- 用户下一步行动清晰。

## 阶段 6：截图 QA

目标：交付前用截图和结构检查发现生成错误。

本阶段必须使用 `failure-patterns.md` 对照检查高频失败模式。

必须检查：

- 状态矩阵主轴是否错位
- 非法组合是否误生成
- 排布能力和展示能力是否混淆
- 空状态是否留白
- 中文是否缺字、空白、裁切
- 状态字段是否联动
- 稳定字段是否被误改
- 组件是否复用原实例
- 补充变体是否在同父组件内
- variant 切换和文案 override 是否真实生效
- auto layout 子节点顺序是否正确
- 权益 / 内容说明是否在活动行之前
- 页面标注是否过度使用画板
- 页面级状态是否完整，不是缩略线框

输出：

- QA 结果
- 修复项
- 无法确认的问题

退出条件：

- 明显视觉错误已修复。
- 高频失败模式已逐项检查。
- 结构顺序已通过脚本或截图确认。
- 如果仍有疑问，已列入需要人工确认的问题。

## 阶段 7：复盘沉淀

目标：把自动稿的问题、用户反馈和调试期标准答案中的差异抽象成可复用规则，沉淀回 skill。正式运行中不要求用户提供标准答案。

适用条件：

- 用户指出自动稿问题
- 自动稿经过多轮修正
- 用户确认某个版本暂时可用
- 调试期用户主动提供手动全情况稿或标准答案

必须输出：

- 差异摘要
- 问题归因
- 可复用规则
- 仅适用于当前案例的规则
- 建议沉淀项和建议更新文件

只有用户明确确认后，才把复盘结论写入对应参考文件；确认前不得声称“已更新文件”。

规则沉淀位置：

- 判断框架：`decision-framework.md`
- 失败模式：`failure-patterns.md`
- 场景拆解：`scenario-decomposition.md`
- 状态分类：`business-state-taxonomy.md`
- 飞猪域模式：`fliggy-domain-patterns.md`
- Figma 写入：`figma-generation-rules.md`
- 质量检查：`quality-checklist.md`
- 内部调试记录：仅在调试 / 回归评测需要时写入隐藏内部目录，不得进入正式流程入口
- 复盘规则：`case-review-patterns.md`
- 能力路由：`capability-routing.md`

退出条件：

- 通用规则已进入规则文件。
- 样本特有内容如需保留，只进入内部调试记录，不出现在正式流程引用中。
- 没有把内部调试资产当作正式生成依赖。
