# 场景拆解方法

## 先拆层级

业务全情况通常不是一个平铺状态列表，而是分层结构。

推荐顺序：

1. 页面级场景
2. 模块级状态
3. 业务对象状态
4. 内容边界状态

## 页面级场景

当以下因素改变页面结构时，应生成页面级场景：

- 单个对象 vs 多个对象
- 单个任务 vs 多个任务
- 单个权益 vs 多个权益
- 单个商家 vs 多个商家
- 是否需要 tab
- 是否需要列表
- 是否需要聚合展示
- 是否需要隐藏某些模块

如果用户的需求明确包含页面级状态，最终 Figma 输出必须保留页面级状态。即使后续进入组件精修，也不能只输出模块级状态矩阵。

示例：

- 商家下有多个挑战：需要 tab 或挑战列表
- 商家下单个挑战：不需要挑战 tab，挑战卡直接前置
- 多权益：需要权益中心横滑或分组
- 单权益：可以简化为单卡片

## 内容边界与布局重排

当状态差异来自“数量减少 / 数量增加 / 内容长度变化 / 信息密度变化”时，不应只机械删除、拉宽或缩放原模块。需要判断该状态下更合理的信息结构。

常见触发：

- 3 个权益变 2 个或 1 个
- 多活动变单活动
- 多任务变单任务
- 多档奖励变单档奖励
- 文案从短句变成长说明
- 图文卡片从多列变单列

判断规则：

- 多列状态通常强调并列比较，适合保持同构卡片。
- 双列状态可以保留纵向卡片结构，但需要重新分配卡片宽度、文案行数和模块高度。
- 单卡状态通常不只是“拉满宽度”，应判断是否需要切换为横向图文、重点卡、单行摘要或更强主视觉。
- 数量减少后，保留哪几个对象不一定是取前 N 个，应根据业务优先级、用户决策价值和主页面强调程度选择。
- 模块内部高度可以随内容压缩，但页面底部固定操作区不一定跟随上移。必须判断底部区域是页面流内容还是固定底 bar。
- 内容边界状态如果改变用户扫描方式，应作为结构变体画稿，而不是只写进矩阵。

## 模块级状态矩阵

当某个模块承载主要业务判断时，应单独生成模块全情况。

常见模块：

- 挑战卡
- 权益卡
- 酒店房型卡
- 会员任务卡
- 优惠券卡
- 订单状态卡

矩阵维度示例：

挑战卡：

- 任务类型：单次任务 / 单进度任务 / 多阶梯任务
- 参与状态：未参加 / 挑战中 / 挑战成功 / 挑战失败结束 / 不可报名

酒店房型卡：

- 库存状态：可订 / 少量库存 / 售罄 / 日期无房
- 价格状态：价格稳定 / 价格变化 / 会员价可用 / 会员价不可用
- 权益状态：优惠可用 / 优惠不可用 / 权益冲突

会员权益卡：

- 用户身份：未登录 / 非会员 / 会员 / 高等级会员
- 权益状态：可领取 / 已领取 / 可使用 / 已使用 / 已过期 / 不可用

酒店详情页会员卡：

- 组件层：标题区 / 权益展示元素 / 权益区详情 / 活动行 / 进度条 / 提示气泡
- 账号状态：未开卡 / 激活账号 / 注册中 / 已开卡 / 会员失效
- 成长状态：无房晚积分 / 有房晚积分 / 即将升级 / 挑战任务中
- 内容边界：4 个权益 / 3 个权益 / 2 个权益 / 1 个权益
- 权益可享性：可享 / 不可享 / 部分不可享

模块级状态不只是状态标签变化。需要同步检查并调整：

- 顶部时间或有效期文案
- 进度值和辅助说明
- 进度文案
- 奖励文案
- 行动点文案
- 主按钮显隐
- 状态标
- 任务卡内部状态
- 权益数量和布局方式
- 权益是否可享的视觉表达
- 标题区右侧行动点、进度条或提示气泡
- 活动行显隐和活动报名状态

同时需要保护不应变化的稳定字段：

- 挑战名称不应因状态变化而改变
- 任务标题不应因状态变化而改变
- 奖品名称、会员等级名称、任务目标名称不应因状态变化而改变
- 酒店品牌、酒店会员等级、权益名称不应因状态变化而改变
- 只有任务类型变化时，任务标题才可以变化
- 判断字段类型时优先看图层名和组件层级。例如 `挑战名称挑战名称` 应始终是挑战名称，即使它当前误写成了“进度9000/12000”这类状态文案，也必须按挑战名称修正。

生成 Figma 前必须确认中文字体可加载。`PingFang SC` 不可加载时，优先改用 `Noto Sans SC` 或当前文件中可加载的中文字体。

## 组合过滤

不能机械生成所有组合。

必须判断：

- 哪些组合业务上不成立
- 哪些组合视觉上没有差异
- 哪些组合只需进矩阵，不需要画稿
- 哪些组合必须画稿

示例：

- 单次任务没有“挑战中”
- 挑战成功后不应展示“立即报名”
- 不可报名应突出原因，而不是展示正常任务进度
- 权益数量变化属于布局边界，4/3/2/1 都应保留；但权益名称仅替换、布局无变化时不必额外画稿。
- 未开卡、激活账号、注册中、会员失效是账号链路状态，不能合并成一个“未开通”。
- 有房晚积分、无房晚积分、挑战任务中、即将升级是成长表达状态，应和账号状态分开建模。
