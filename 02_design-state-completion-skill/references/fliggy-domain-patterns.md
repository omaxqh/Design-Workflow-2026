# 飞猪业务域模式

本文件用于把 `business-state-taxonomy.md` 中的通用业务状态映射到飞猪 App 的页面、模块和 Figma 生成语境。

使用边界：

- 不重复维护通用状态清单；状态类型、判断变量和不适用规则先读 `business-state-taxonomy.md`。
- 本文件只补充飞猪语境下的页面结构、模块承载、组件拆分、字段联动和必须画稿判断。
- 视觉规范和组件规范应从 `fliggy-design-gui` skill 中读取。

## 规范来源

涉及飞猪页面时，应优先读取：

- `fliggy-design-gui` 的 `SKILL.md`
- design foundations
- 酒店域页面规范
- 会员域页面规范
- 相关页面 example 或 spec

注意：

- 本 skill 不生成 HTML，不沿用 `fliggy-design-gui` 的 HTML 交付方式。
- 本 skill 使用其中的设计规范、业务域约束和页面模式，用于 Figma 补全。
- 对已有主页面 Figma，应优先继承页面级框架和组件，不强行套用仓库 HTML 模板。

## 酒店域重点

状态来源：先使用 `business-state-taxonomy.md` 中的酒店域、会员、权益、交易和内容边界状态。

飞猪酒店域落地重点：

- 酒店详情页、房型 rate、普通 rate、酒店会员卡和预订链路通常同时受库存、价格、会员权益、取消政策和用户资格影响。
- 同一状态如果改变价格、库存、权益可用性、底部 CTA、模块显隐或用户下一步动作，应进入必须画稿列表。
- 只影响底层接口或平台基础异常的状态，默认只在矩阵中标注由平台基础规范覆盖。

设计判断：

- 价格、库存、权益、取消政策和会员资格应联动到用户可见字段，不只改状态标签。
- 涉及底部 CTA、模块显隐、折叠、替换或大幅结构调整时，必须画稿。
- 如果页面结构需要大幅调整，标为结构变体。

### 酒店详情页会员表达模块

用于酒店详情页中展示“通过飞猪 F 等可直通酒店会员，以及开通后可享酒店会员权益”的二官卡 / 会员卡模块。

核心业务变量：

- 开通状态：未开卡 / 激活账号 / 注册中 / 已开卡 / 会员失效。
- 成长状态：无房晚积分 / 有房晚积分 / 即将升级。
- 挑战状态：挑战任务中。
- 权益数量：4 个 / 3 个 / 2 个 / 1 个。
- 权益可享性：可享 / 不可享。
- 活动行状态：有活动 / 报名成功 / 无活动。

设计判断：

- 该类模块必须同时补“子组件变体”和“页面中模块状态”。
- 子组件变体包括标题区、权益展示元素、权益区详情、进度条、提示气泡、活动行。
- 页面中模块状态用于验证子组件组合后的真实详情页展示，不应只停留在组件库。
- 权益数量变化会影响布局密度和高度，4/3 个权益通常使用多列权益块，2/1 个权益可转为单行权益表达。
- 未开卡、激活账号、注册中、会员失效会改变标题区和行动文案，必须单独画稿。
- 有房晚积分、无房晚积分、挑战任务中、即将升级会改变标题右侧的进度、进度条和提示气泡。
- 无活动状态会影响活动行显隐和模块高度，必须在页面级模块状态中列举。

## 会员域重点

状态来源：先使用 `business-state-taxonomy.md` 中的会员、权益、活动 / 挑战和内容边界状态。

飞猪会员域落地重点：

- 会员身份变化通常会联动页面头部、权益卡、任务模块、价格表达和 CTA。
- 会员模块嵌入酒店详情、交易或活动页面时，应同时检查组件内部状态和页面内承接状态。
- 酒店会员账号开通链路中的“未开卡、激活账号、注册中、已开卡、已失效”是不同业务节点，不是同一状态的文案替换。
- 成长进度、房晚 / 积分、挑战任务和即将升级应与账号开通状态分开建模。

设计判断：

- 会员身份、权益领取 / 使用 / 失效、成长任务阶段结果若影响用户决策，应画稿。
- 仅通用空态和网络失败不默认画。
- 嵌入交易或详情页的会员模块，必须同时检查组件内部状态和页面内承接状态。
- 账号开通链路和成长进度链路分开建模，不合并成一个“未开通 / 已开通”二元状态。

### 酒店 rate 权益强化表达

用于酒店详情页中，围绕房型 rate、普通 rate 等不同 rate 组件强化 88VIP、会员权益、取消政策、早餐等利益点表达。

核心业务变量：

- rate 类型：房型 rate / 普通 rate。
- 权益类型：免费早餐 / 无忧退 / 会员专享价 / 积分或里程抵扣。
- 表达阶段：权益强化前 / 权益强化后。
- 承载字段：早餐字段 / 退改政策字段 / 运营坑标 / 服务标签 / 价格利益点。

设计判断：

- 如果主页面只给出某一种 rate 类型的 before / after，应主动检查同域是否存在其他 rate 组件，并判断权益表达是否需要迁移补齐。
- 权益强化不是整卡重画，优先复用目标 rate 的原组件实例，再联动改写承载字段。
- 免费早餐类权益通常优先落在早餐字段，其次同步运营坑标。
- 无忧退类权益通常优先落在退改政策字段，其次同步运营坑标或服务标签。
- 如果无忧退权益后态的业务确认是“运营坑标保持 F4 无忧退”，则强化表达应落在退改政策文案，不要把运营坑标强行切成 88VIP 样式。
- 运营坑标、早餐、退改政策、服务标签、价格利益点是可变字段；酒店名、店铺名、价格结构、按钮样式是稳定字段，除非需求明确要求变化。
- 房型 rate 和普通 rate 的布局密度不同，不能直接复制房型 rate 画法到普通 rate；应在普通 rate 组件内寻找等价承载点。

## 活动权益重点

状态来源：先使用 `business-state-taxonomy.md` 中的活动 / 挑战、权益 / 优惠和时间窗口状态。

飞猪活动权益落地重点：

- 活动承接页要先判断单活动 / 多活动、单商家 / 多商家、是否需要 tab 或列表，再进入挑战卡矩阵。
- 活动状态不能只补成功 / 失败，应覆盖参与前、参与中、结果后、资格不满足和时间窗口变化。
- 奖励发放、权益领取、任务进度和报名资格如果影响 CTA 或利益点表达，应进入必须画稿列表。

设计判断：

- 活动承接通常需要完整状态补全，不能只补成功 / 失败。
- 活动结果、任务进度、奖励发放或资格变化会改变主 CTA 或利益点表达时，必须画稿。
- 如果同一商家下可承载多个活动，应先判断单活动和多活动的页面级结构差异。
