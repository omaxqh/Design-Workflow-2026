# SmartDesigner — 智能设计决策助手

> 理解产品需求，结合设计原则，输出可直接驱动设计执行的 DRD（Design Requirements Document）。

---

## 触发条件

当用户提供产品需求文档（PRD）、设计稿截图、或业务背景描述，并希望获得设计方向建议时使用。

触发词：`/smart-design`、"帮我分析设计方向"、"审阅这个方案"、"设计决策"

---

## 项目结构

```
SmartDesigner/
├── Step1_放入资料/          ← 按项目文件夹放入资料
│   ├── 示例Project/
│   │   ├── Prd/          # 产品需求文档
│   │   ├── 沟通纪要/     # 会议纪要、评审记录
│   │   └── 用研资料/     # 用户研究报告、调研数据
│   └── INPUT.md
├── Step2_用SKILL分析/      ← 本文件所在，定义分析流程
│   └── SKILL.md
├── Step3_查看分析结果/     ← 按项目名新建文件夹，输出 .md 文档
│   └── 示例Project结果/
├── DesignPrinciple.md      ← 设计原则知识库（通用原则/反模式/评分）
├── BussinessExp.md         ← 业务知识库（行业经验/业务规则/项目决策）
└── Reference               ← 参考方案
```

支持的输入类型：
- `.md` 文档（PRD、需求描述、会议纪要、用研报告）
- `.png` / `.jpg` 图片（设计稿截图、竞品截图）
- `.txt` / `.pdf` 文本（沟通记录、数据报告）
- *资料类型可拓展*

## Pipeline（分步执行）

### Step 1：需求理解

**目标**：从用户输入中提取结构化需求信息。

**输入来源**（按优先级）：
1. 用户直接提供的 PRD 链接（如 alidocs.dingtalk.com、yuque.antfin.com 等）→ 使用 **IDE 内置 Browser Agent** 打开并提取文档内容
2. 本地文件：读取 `Step1_放入资料/[项目名]/` 中所有子文件夹的资料
3. 用户口述的需求描述

> **PRD 链接处理（默认路径，不得绕过）**：
>
> **Step A · 默认**：调用 Qoder/IDE 内置 **Browser Agent**（subagent_type=Browser）打开链接。IDE 浏览器共享用户 SSO 登录态，内网文档（aliyuque.antfin.com / alidocs.dingtalk.com / yuque.antfin.com）大多数情况可直接读取。
>
> **Step B · 遇登录拦截时**（页面空壳 / 跳 SSO / 401）：Browser Agent 在 IDE 浏览器窗口中**暂停等待用户现场完成登录**，登录完成后 Browser Agent 再次读 DOM 抽全文。全程无需用户导出 cookie 或复制粘贴。
>
> **Step C · 仅当 Browser Agent 整体不可用 / 登录持续失败时** → 中断并让用户二选一：
> - C1. `/setup-browser-cookies` 导入 aliyuque/antfin cookie 后重试（适合需要反复抓取的场景）
> - C2. 用户手动粘贴 PRD 全文（最后兜底）
>
> **禁止行为**：
> - 不得一上来就用无登录态的外部 fetcher（fetch_content 对内网返回 `loading` 即为无登录态）尝试后立刻升级到 C1/C2
> - 不得静默降级；从 A 到 B 到 C 任何一级切换都必须显式告知用户原因
> - 不得跳过 A 直接执行 C（参见：内网知识库访问失败交互规范）

从获取的资料中提取以下信息：

```
- 行业 & 业务：这是什么行业的什么业务？
- 业务背景：当前业务现状是什么？
- 目标用户：谁在用？用户画像是什么？
- 用户痛点：用户遇到了什么问题？（按优先级排序）
- 产品方案：产品提出了什么解决方案？
- 期望目标：希望达成什么效果？
```

**输出**：结构化需求摘要（打印给用户确认）

### Step 2：用户研究

**目标**：构建目标用户画像，理解用户心智。

**执行方式：通过「用户画像生成器」在线工具生成**

工具地址：`https://aistudio.alibaba-inc.com/aiWorktable#/aistudio/webPage?appCode=zgewbVfNDRr`

执行顺序：
1. 使用浏览器打开上述工具页面（Qoder 内置浏览器已有登录态可直接使用）
2. 如访问失败（未登录/cookie 过期），**必须提示用户**：
   ```
   用户画像生成器访问失败（原因：[具体错误]）。
   请选择：
   A. 执行 /setup-browser-cookies 导入内网登录态，然后重试
   B. 跳过在线工具，基于 PRD 信息独立推导用户画像
   ```
3. 访问成功后，按以下步骤操作页面：
   
   **a. 展开变量面板**：点击页面底部的「展开变量」按钮（如已显示「收起变量」则已展开）
   
   **b. 填写4个表单字段**：
   - **行业分类** ← Step 1「行业 & 业务」
   - **用户是谁** ← Step 1「目标用户」
   - **用户遇到了什么问题** ← Step 1「用户痛点」
   - **知识库索引关键词** ← 从 Step 1 提取 3-5 个核心关键词
   
   **c. 发送消息**：在底部消息输入框（placeholder「请输入」）中输入 `开始分析`
   
   **d. 点击发送**：用**精确 DOM 选择器**锁定发送按钮（页面底部工具栏有 3-5 个同类 `.chat-input-icon-btn`，上传/语音/发送都是这个 class，**只能靠内部 SVG 的 `use` href 区分**）：
   
   ```css
   /* 首选 — 最稳定，直接锁发送键 */
   use[*|href="#icon-chat-run-btn"]
   
   /* 备选 — 带父级容器 */
   div.chat-input-icon-btn:has(use[href="#icon-chat-run-btn"])
   ```
   
   DOM 形态（供 Browser Agent 参考）：
   ```html
   <div class="chat-input-icon-btn">
     <i class="next-icon next-medium">
       <svg class="next-icon-remote">
         <use xlink:href="#icon-chat-run-btn"></use>
       </svg>
     </i>
   </div>
   ```
   
   ❌ **禁止**的错误定位方式（血泪教训）：
   - 不要用「▶ 小飞机图标」这种人工描述找 — Browser Agent 会扫遍整页
   - 不要用 `.chat-input-icon-btn:last-child` 或 `:nth-child(N)` — DOM 顺序会变
   - 不要用 `data-aplus-clk="x9_xxx"` / `data-spm-anchor-id` — 埋点 ID 每次刷新都变
   
   **e. 等待生成**：等待 30-60 秒，AI 会返回完整的用户画像报告
   
4. 提取生成的用户画像内容（包含：典型画像、用户原语、情景描述、报告出处）
5. 将生成结果整合到本步骤输出中

> **注意**：报告生成需要时间，不要重复点击「展开变量」或重复发送消息，耐心等待结果完整输出。

**用户研究输出**：

基于用户画像生成器结果 + Step 1 信息 + 知识库参考资料（如可用）：

1. 生成 2-3 个典型用户画像（含年龄、行为特征、核心诉求、用户原语）
2. 描述每个用户在当前场景下的具体使用情景
3. 列出用户的认知习惯和决策偏好
4. 标注用户最敏感的体验触点

### Step 3：PRD 审核 + 设计机会点提取

**目标**：用设计原则和业务知识**审视 PRD 方案在各体验维度上的质量**，找出方案在链路、文案、交互、信息层级等环节对用户体验的损伤，提炼设计机会点。

> **核心理念**：PRD 方案的业务方向通常是合理的，但具体落地时往往会在链路、文案、交互、信息层级等维度损伤用户体验。设计师的价值是识别这些损伤点，在满足业务诉求的前提下提出更合理的设计方案。有时是改一句文案，有时是调整信息层级，有时才需要完全不同的解法。

**输入**：
- Step 1 结构化需求摘要
- Step 2 用户画像 + 行为洞察
- `../DesignPrinciple.md`（通用设计原则 + 反模式清单）
- `../BussinessExp.md`（行业经验 + 业务规则）

**执行——三层分析法**：

1. **读取双文件**：先读 `BussinessExp.md` 理解行业上下文和业务约束，再读 `DesignPrinciple.md` 获取评估工具（四维度 + 反模式 + 心理学原则）

2. **第一层 · 提取真实用户问题**：从 PRD 每条需求中抽离出「用户到底需要解决什么问题」（业务诉求，通常是合理的）

3. **第二层 · 诊断方案缺陷**：用三视角审视 PRD 提出的具体方案在哪些维度损伤了用户体验：

   | 审视视角 | 评估工具 | 审核问题 |
   |---------|---------|----------|
   | **用户视角** | Step 2 用户画像 | 目标用户在真实场景中会怎么理解/使用这个方案？哪里会困惑/流失？ |
   | **原则视角** | DP 四维度 + 反模式 | 方案在链路连贯性、信息完整性、文案可读性、交互好用性哪个维度有伤害？命中了哪条反模式？ |
   | **业务视角** | BussinessExp | 方案是否违背行业经验（如旅行场景价格透明、会员场景确定性偏好）？ |

4. **第三层 · 提出设计机会点**：针对诊断出的每个损伤点，提出设计介入方向——**不一定是完全不同的解法**，更常见的是：
   - 文案损伤 → 改文案就够
   - 层级损伤 → 调信息架构
   - 链路损伤 → 优化流程/减步骤
   - 多维度损伤 → 可能需要替代方案

5. **输出 PRD 审核表**（必出）：

   ```
   ## PRD 审核表
   
   | # | PRD 需求条目 | 真实用户问题 | ✅ 方案合理之处 | ⚠️ 体验损伤点 | 损伤维度 + DP/BE 依据 | 💡 设计机会点 |
   |---|-------------|------------|---------------|-------------|----------------------|-------------|
   | 1 | [PRD原文] | [用户实际要解决的问题] | [业务方向合理] | [具体哪里伤害体验] | [链路/文案/交互/信息] + [DP条款] | [设计可以做什么] |
   
   **审核统计**：
   - 方案无体验损伤，可直接采纳：[N] 条
   - 方案有体验损伤，需设计介入：[N] 条
   - 设计机会点总计：[N] 个
   ```

6. **关键区分——产品需求 vs 设计方案**：
   - **产品需求**（PRD 该管的）：做什么功能、面向什么人群、达成什么目标
   - **设计方案**（设计师该管的）：怎么展示、用什么文案、什么交互、什么层级、什么流程
   - PRD 中如果越界到「用什么按钮」「文案写XX」→ 审核表中标注 `⚠️ 产品越界`，设计师有权重新定义

**硬闸**：未输出 PRD 审核表 → 不得进 Step 3.5

### Step 3.5：设计师选择门（六维分类）

**目标**：在出 DRD 之前，**让设计师做选择**——不是所有问题都要出设计稿，有些改一句文案就解决了。

> **核心理念**：AI 不要吭哧吭哧出一堆没用的完整方案。根据损伤点的性质匹配最合适的设计思维深度，文案能解决的就用文案，只有问题定义本身有偏时才需要重构。

**输入**：Step 3 PRD 审核表中所有设计机会点

**执行**：

1. **为每个机会点匹配设计思维深度**（六维分类，从深到浅）：

   | 维度 | 思维深度 | 适用场景 | 产出物 | 成本参考 |
   |------|---------|---------|--------|----------|
   | **D1 · 问题重构** | PRD 的问题定义本身就偏了 | 用户的真实痛点与 PRD 描述的问题不一致；需求解决的是表层症状而非根因 | 重新定义问题 + 新方案 | 3-5天 |
   | **D2 · 方案替代** | 问题对，但有更好的解决路径 | 「更快的马→汽车」；多步操作可被自动化替代；新增页面可被原地变态替代 | 替代方案 + 对比说明 | 2-3天 |
   | **D3 · 流程优化** | 方向对，但交互流程/步骤可改善 | 链路多了不必要的步骤；操作顺序不符合用户心智；决策路径断裂 | 流程图对比 + Mermaid | 1-2天 |
   | **D4 · 信息重构** | 内容对，但层级/优先级/表达方式有问题 | 重要信息被埋没；信息分组不符合认知习惯；主次不分 | 信息架构图 + 层级说明 | 0.5-1天 |
   | **D5 · 组件调整** | 结构对，但某个组件的视觉/交互可改善 | 按钮强调错位；卡片样式不符合场景；状态反馈不足 | 组件 diff + Token diff | 2-4h |
   | **D6 · 文案修正** | 都对，只有措词不符合用户心智 | 术语、歧义、过长、无利益点、压力文案 | 文案对照表（原→推荐→避免） | 0.5-1h |

   > **匹配规则**：从 D6 往 D1 查，优先匹配最浅层的维度。只有当浅层确实无法解决问题时，才向更深层匹配。

2. **每个机会点给 2-3 个解法选项**（从不同维度给出），让设计师选：

   ```
   🚦 设计师选择门（不选不出 DRD）
   
   机会点 1：[损伤描述]
     ○ D6 文案修正：把「XX」改为「YY」 → 0.5h
     ○ D5 组件调整：加一个状态标签强化反馈 → 3h
     → 推荐：D6 · 理由：文案本身是问题根因，改词即可解决
   
   机会点 2：[损伤描述]
     ○ D4 信息重构：调整卡片内信息优先级 → 4h
     ○ D3 流程优化：合并两步为一步 → 1天
     ○ D2 方案替代：用“自动推送+原地变态”替代“新增进度页” → 2天
     → 推荐：D4 · 理由：信息层级调整即可解决，不需改流程
   ```

3. **推荐规则**：

   **3.1 默认最浅层优先**（D6 > D5 > D4 > D3 > D2 > D1）
   - 浅层无法解决时才升维，每升一级必须说明「为什么更浅层解决不了」
   - 如果 D6 就能解决，**禁止同时推荐 D2/D1**

   **3.2 何时必须升维（判定清单）**：

   | 从 → 到 | 必须升维的信号 |
   |---------|--------------|
   | D6 → D5 | 改完文案还是看不懂 / 文案改了但位置不对 / 文案无承载体 |
   | D5 → D4 | 组件调完仍信息过载 / 该重要的还是不重要 / 用户找不到关键入口 |
   | D4 → D3 | 信息都对但用户完成任务路径仍断裂 / 必经节点缺失 / 步骤顺序颠倒 |
   | D3 → D2 | 流程无论怎么优化都冗长 / 用户心智上就不该有这一步 / 同场景竞品已有更优范式 |
   | D2 → D1 | 多个替代方案都被用户拒绝 / 用户真实痛点与 PRD 描述完全不符 / 解决了这个还有下一个同源问题 |

   **3.3 升维质疑机制**：
   - D1/D2 占比 > 30% → 主动质疑「是否高估了问题深度？大部分问题真的需要重构吗？」
   - 单个机会点推荐 D1 但没用户画像支撑 → 拒绝推荐，回退 D2
   - 同一 PRD 推荐 > 1 个 D1 → 提示「重构应聚焦单一核心问题，多个 D1 是产品阶段问题」

   **3.4 推荐示例对照**：

   ✅ **正确示例 A**（文案问题用 D6 解决）：
   ```
   机会点：订单详情的「申请进度」用户不理解是什么
     ○ D6 文案修正：改为「千问正在帮您申请免费升房」→ 0.5h
     ○ D5 组件调整：加进度条组件 → 4h
     → 推荐：D6
     → 理由：用户困惑的根因是「进度」一词抽象，换成人话即可；加进度条是解决表达不是解决理解
     → 为什么不推荐 D5：组件成本 8×，且没有真实进度数据支撑，会造成新的体验负担
   ```

   ✅ **正确示例 B**（链路问题必须升到 D2）：
   ```
   机会点：PRD 要求新增「升房进度页」让用户主动查询
     ○ D6 文案修正：把进度页文案优化 → 0.5h
     ○ D4 信息重构：把进度信息放到订单详情 → 4h
     ○ D2 方案替代：用「主动推送+订单详情原地变态」替代新增页面 → 2天
     → 推荐：D2
     → 理由：用户真实诉求是「知道结果」而非「主动查进度」；新增页会引入二次跳转（违反 DP 四.链路连贯性）
     → 为什么不推荐 D4：信息搬到订详仍需用户主动进入，没解决「主动查询」这个伪需求
     → 为什么不推荐 D6：方案层面就错了，改文案是给错误方案画皮
   ```

   ❌ **错误示例 A**（浅层就能解决却推荐深层）：
   ```
   机会点：按钮文案「确认」用户理解为「提交申请」
     ○ D6：改为「提交申请」→ 0.5h
     ○ D2：重设计整个申请流程 → 2天
     → 推荐：D2  ← ❌ 违反 3.1，D6 已能完全解决
   ```

   ❌ **错误示例 B**（升维理由不成立）：
   ```
   机会点：权益卡没被用户注意到
     ○ D5：加金色描边 + 徽章 → 2h
     ○ D1：重新定义「权益展示」这个问题 → 5天
     → 推荐：D1  ← ❌ 违反 3.3，无用户画像数据支撑，属于过度重构
   ```

   **3.5 跨维度组合规则**：
   - 一个 PRD 通常是 D6-D4 为主（60%+）、D3-D2 少量（30%）、D1 罕见（< 10%）
   - 同一机会点禁止推荐跨 2 个维度的选项（如 D6 + D2 并列），差距太大说明问题定义不清
   - 多个机会点之间可以跨维度组合（如 3 个 D6 + 1 个 D3）

4. **等待设计师逐条选择**（不选不出 DRD）

**硬闸**：
- 未给出选项 → 不得进 Step 4
- 未等设计师选择 → 不得进 Step 4
- D1/D2 占比 > 30% → 主动质疑「是否高估了问题深度？大部分问题真的需要重构吗？」

**六维与 Step 5.5 L0-L6 的关系**：
- D1-D6 是「设计思维深度」，决定用什么类型的思考解决问题
- L0-L6 是「产出物颗粒度」，决定最终交付什么格式的文件

### Step 5.5：设计范围界定（Design Scope Definition）

**目的**：PRD 面向全团队（产品/技术/运营/设计/法务），描述的是「要做什么业务」，不是「要设计什么」。本步骤把 DRD 中每一条需求映射到「设计干预层级」，**决定要不要出稿、出什么颗粒度的稿**。不做分层，LLM 会把一句文案修改也画成 500 行 HTML——这是 AI 味的结构性根因。

**核心原则**：
- **产物颗粒 = 干预层级**：L3 需求不得产出 L4 稿，L1 需求不得产出 L3 稿
- **80/20 规则**：真正需要 L4+ 的往往 ≤ 20%，大量 AI 味来自把 L1-L3 做成了 L4+
- **可不出稿**：L0/L1 不进 Step 6 HTML 生成，直接出文档即可交付

#### 设计干预分层模型（L0-L6）

| 层级 | 干预颗粒 | 判断信号 | 正确产物 | 禁出产物 |
|------|---------|---------|---------|----------|
| **L0 · 零设计** | 后端规则/业务逻辑变更，用户不感知视觉变化 | “扩大人群/调整阈值/新增权益判断”且无 UI 改动 | 一句话说明「本条无需设计」+ 给出理由 | ❌ HTML ❌ 文案表 |
| **L1 · 文案层** | 只改一个词/一句话，不动结构、位置、样式 | PRD 里写「按钮文案改为 XX」「提示文案调为 XX」 | 文案对照表（推荐 vs 避免 vs 原文案）+ 位置标注 | ❌ HTML ❌ 组件图 |
| **L2 · Token 层** | 颜色/字号/间距/图标微调 | 「将 X 颜色改为 Y」「统一 XX 为胶囊」「得到新图标」 | Token diff 表 / 单图标 SVG+尺寸规范 | ❌ 整页稿 |
| **L3 · 组件层** | 在现有页面新增/修改某个组件，其他区域不动 | 「在 X 页添加一个 Y 标签/卡片/按钮/状态头部」 | **组件 diff** （前后对比片段 + 插入位置标注 + 尺寸） | ❌ 整页 HTML |
| **L4 · 页面层** | 重排一个页面的信息架构，大部分区域调整 | 「重设计 X 页」「调整 X 页的信息层级」 | 完整页面 HTML + 状态枚举 | ❌ 跨页链路图 |
| **L5 · 新界面** | 全新浮层/页面/弹窗 | 「新增 X 弹窗」「添加 X 浮窗」 | 完整 HTML + 状态枚举 + 快捷规则 | — |
| **L6 · 全链路** | 跨 ≥3 页的新流程 | 「新增开卡流程」「重构 XX 链路」 | Mermaid 链路 + 各节点 HTML 集 + 传递规则 | — |

#### 分层判断防误盒

- **分不清 L2 还是 L3？** → 只改样式值=L2；增减元素=L3
- **分不清 L3 还是 L4？** → 所动区域占页面 <30%=L3；≥ 30% 或影响其他区域=L4
- **分不清 L4 还是 L5？** → 原页结构还在=L4；全新容器=L5
- **分不清 L5 还是 L6？** → 单触点自闭环=L5；需跨页传递状态=L6
- **遇到 L0/L1 却被要求出稿** → 主动暴露「此条为 LX，无需 HTML，建议交付格式为 XX」询问用户是否坚持

#### 分层判断双维度信号（A 类型 + B 约束）

**分层不是「看一眼就知道」，必须走双维度信号链。A 维度出「初评」，B 维度出「修正」与「约束标签」，两者合成才是「最终层级」。**

##### A · 类型信号（WHAT）— 决定初评

| 信号 | 判断问题 | 信号映射 |
|------|---------|---------|
| **A1 动词信号** | PRD 用了什么动词？ | 调整规则/扩大人群=L0；修改文案=L1；调整颜色/图标=L2；新增 XX 标签/按钮=L3；重设计 X 页=L4；新增 XX 弹窗=L5；新增 XX 流程=L6 |
| **A2 影响范围** | 改动占现有页面多少比例？ | <10% 单元素=L2/L3；10-30% 局部=L3；30-70% 主结构=L4；>70% 或全新容器=L5；跨≥3页=L6 |
| **A3 用户感知** | 用户能感知的最小变化是什么？ | 无视觉变化=L0；一个词不同=L1；颜色/字号不同=L2；多/少了元素=L3；页面布局变=L4；出现新界面=L5 |
| **A4 DRD 触点描述** | DRD 第六章「设计规格」怎么描述该触点？ | 文案变更=L1；Token 调整=L2；新增组件=L3；页面重构=L4；新页/新弹窗=L5；新流程=L6 |

**A 初评规则**：4 类信号必须至少 2 类一致才能定级；不一致 → 暴露冲突并问用户。

##### B · 约束信号（WHY/HOW）— 修正 ±N 级 + 加标签

| 信号 | 判断问题 | 对层级的影响 | 加什么标签 |
|------|---------|---------------|----------|
| **B1 优先级** | P0 必做 / P1 可做 / P2 长期 | P2 强制 -1 级或暂缓 | `[暂缓]` `[P2 降级]` |
| **B2 业务权重** | 是否核心转化漏斗 / 大促节点 | 核心 +1 级精度；边缘 -1 级 | `[核心交易]` `[边缘触点]` |
| **B3 规范沉淀** | 一次性专项 / 进 Design System | 系统级 → 强制追加 Token + 组件规范文档 | `[系统Token]` `[一次性专项]` |
| **B4 改动安全** | 现有规范是否不能大改 / 是否多业务共用核心组件 | 不能大改 → 强制「贴外侧加新组件」而非「改老组件」 | `[不改老组件]` `[贴附加]` |
| **B5 资源预算** | 几人天 / 几个工期 | 1人天=L1/L2 上限；3人天=L3；1周=L4/L5；>2周=L6 | `[1人天]` `[3人天]` `[>2周]` |
| **B6 风险/可逆性** | 灰度 / 全量 / 可回滚 | 不可逆 → Token 命名带 v2 + 强制灰度方案 | `[灰度]` `[全量]` `[Token v2]` |
| **B7 法务/合规** | 是否涉及金额/隐私/广告法/承诺话术 | 文案标注「法务锁定」，不可创作 | `[法务锁定]` `[待法务确认]` |
| **B8 跨端一致** | 单端 H5 / 多端同步 | 多端 → 必须出 Token 层，不能只出单端 HTML | `[三端Token]` `[多端对齐]` |

##### 合成公式

```
最终层级 = clamp( A 初评 ± B1∼B8 修正系数, L0, L6 )
约束标签 = B1∼B8 命中条目并入产物要求
```

##### 示例：T3D 退款完成态

- A1 动词（DRD「展示到账完成态」）→ L3；A2 范围（仅顶部状态区 <20%）→ L3；A3 感知（状态头部多了加速带）→ L3；A4 DRD（列为「订单详情状态变更」）→ L3 → **A 初评 = L3**
- B1 P0 +0；B2 不在核心漏斗边缘 +0；B3 88VIP 是 IP 需进 DS → **+系统Token**；B4 现有状态头部是核心组件 → **贴附加不改老**；B5 1人天 → L3 上限；B6 全量 → Token 命名 `--88vip-rapid-*`；B7「2小时到账」=承诺话术 → **法务锁定**；B8 三端一致 → **三端Token**
- **最终 = L3 + [系统Token / 贴附加不改老 / 法务锁定 / 三端Token]**
- 产物：1 个组件 diff（状态头部上方贴装加速带）+ 1 份 Token diff（`--vip-rapid-*`）+ 1 份文案锁定表

#### 分层表输出格式（必输出）

```markdown

| # | 需求条目 | A1动词 | A2范围 | A3感知 | A4 DRD | **A初评** | B1优先 | B2权重 | B3规范 | B4安全 | B5预算 | B6可逆 | B7法务 | B8跨端 | **最终层级** | **约束标签** |

**进入 Step 6 的条目**：仅 L3 及以上，共 [N] 条
**仅出文档的条目**：L0/L1/L2 共 [N] 条，直接在 Step 5.5 交付
```

#### Step 5.5 直接交付（L0/L1/L2 不进 Step 6）

- **L0 条目**：在范围表旁边写一句话说明「本条无需设计，理由：XX」
- **L1 条目**：在本小节输出「文案对照表」：`| 位置 | 原文案 | 推荐文案 | 避免文案 | DP 条款 | 法务锁定 |`
- **L2 条目**：在本小节输出「Token diff 表」：`| Token名 | 原值 | 新值 | 影响范围 | 跨端同步 |`
- **带「系统Token」标签的 L3+ 条目**：额外补交「Token diff 表」+「组件规范文档」
- **带「法务锁定」标签的条目**：文案位置高亮标红，Agent 不得改写
- **带「贴附加不改老」标签的条目**：产物中明确插入位置为「原组件之上/之外」，禁改原组件 DOM

**硬闸规则**：
- 未输出分层表 → 不得进 Step 6
- A 初评 4 信号一致度 <2/4 → 必须暴露冲突并询问用户，不得默认定级
- B 维度任何一项未填（打「未知」） → 必须问用户，不得凭经验默认
- L0/L1/L2 条目被错误带入 Step 6 → Step 6.1 触点清单阶段打回
- 分层统计后 L4+ 条目占比 >50% → 向用户主动质疑「是否高估了干预颗粒？」

#### 分层确认门（必须等待用户逐条回应，才能进 Step 6）

输出分层表后，Agent 必须主动询问用户，请逐条回应（以下 6 个动作之一）：

- ✅ **维持**：接受 LLM 建议层级 + 标签
- ⬇️ **降级**：从 L[N] 降为 L[N-1]，理由：___
- ⬆️ **升级**：从 L[N] 升为 L[N+1]，理由：___
- ✂️ **拆分**：这一条实际是 N 个独立需求，分别为 L[X]/L[Y]
- 🤝 **合并**：与第 [N] 条合并，统一为 L[X]
- ⏸️ **暂缓**：本期不做，移入长期目标
- 🏷️ **加标签**：手动补 B 维度约束标签（如 LLM 未识别到「跨端」/「法务」等）

🔒 **未收到逐条确认 → 不得进 Step 6**；允许「全部接受」作为一次性回应。

### Step 6：方案设计（设计稿生成）

**目标**：基于 DRD 设计规格 + Design_DNA 组件规范，生成可执行的 HTML 设计方案。

**核心原则**：
- **层级匹配（P0）**：Step 6 产物颗粒必须≤ Step 5.5 声明的层级；L3 条目不得产出整页 HTML，只允许组件 diff；L0/L1/L2 条目禁止进入本 Step
- **原则先行（P0）**：Step 6 开始前必须读 `../DesignPrinciple.md` 全文，所有设计决策显式绑定 DP 章节/条款；DP 未覆盖的决策维度必须在 6.2 暴露并询问用户，不得凭经验默认
- **DNA 负责「像」，DP 负责「好」**：Design_DNA 约束视觉体系（token/栅格/字体），DesignPrinciple 约束设计质量（层次/文案/交互/心理）；两者缺一，输出必为 AI 套路
- **优先复制模板**：如果触点对应的 Design_DNA 中有模板 HTML（如 `hotel-order.html`、`hotel-booking.html`），必须基于副本增量修改；如果没有对应模板，则新建 HTML 但必须遵循对应 DNA 的 SKILL.md 规范
- **禁止臆断触点**：仅实现 DRD 中明确定义触发条件的页面，不自行新增
- **不确定先问**：任何 DRD 未明确、且无法从 PRD 推断的设计决策，必须先询问用户
- **三向可追溯**：每个 UI 元素必须能追溯到（A）PRD/DRD 行号、（B）DP 章节条款、（C）DNA 组件规范；任一缺失即为自作主张
- **决策闭环完整**：任何决策型界面必须满足用户决策 5 要素（身份/动作/代价/边界/行动）
- **极简有底线**：极简不能删除否定权、决策要素、情感语言（见 6.2 极简反模式 3 红线）

#### 6.0 三向追溯字典（PRD/DRD + DesignPrinciple + DNA）— **规则 + Demo 合并输出**

**目的**：LLM 从长文档提取到短设计稿必然有概率性遗漏。本步骤把「单向提取」升级为「三向可验证追溯」——既追溯业务（PRD/DRD），也追溯原则（DP），再追溯视觉体系（DNA）。

**核心规则（❗ 血泪教训）**：
> **字典不得以抽象规则表形式交付**。每一条字典条目必须**同步附带本项目的真实 Demo 片段**（文案原句 / ASCII mini 线框 / HTML snippet / 色块效果），让用户**看 Demo 直接拍板**，而不是看抽象规则猜落地。
>
> **判定标准**：如果用户读完字典无法 5 秒内回答「这条在本项目长啥样」，该字典不合格，必须重写。

**执行**：在进入 6.1 之前，并行建立三类字典。

##### 字典 A：PRD/DRD 业务词 → **真实文案/数字 Demo**

每行必须给**本项目真实出现的完整文案/数字**（非关键词清单），用户看 Demo 就能判断对错。

```
| 类别 | 本项目 Demo（真实文案/数字原句） | 出处 |
|------|--------------------------------|------|
| 身份/资格 | 「希尔顿金卡」「飞猪 F4 会员」「万豪珺品集团卡」（完整称谓，非「会员」缩写） | PRD 4.1 |
| 时效/边界 | 「入住前 2 天 9:00 开启预约」「24h 内告知」「延退到 16:00」 | PRD 4.7 |
| 金额/数量 | 「立省 ¥187」「¥450-650/万分」「2 种可升房型」 | DRD §三 |
| 状态文案(完整句) | 「已提交升房申请 · 酒店 24h 内告知结果 · 提交于 5.10 09:12」 | DRD §六 态2 |
| 原文引用 | 「✓ 已升级至豪华大床房 · 直接出示身份证」 | DRD §三 |
| 反面禁用完整句 | ❌「升房处理中,请耐心等待」 ❌「亲,升房通道已开启!」 | DRD 文案避免列 |
```

##### 字典 B：DP 原则 → **❌反例 Demo ↔ ✅本项目 Demo 对照**

每条 DP 条款必须同步给「反例 Demo」和「本项目要做成啥样的 Demo」，让用户看对照就能判断。

```
| DP 条款 | ❌ 反例 Demo（违反会发生的事） | ✅ 本项目 Demo（实际要长这样） | 落地触点 |
|---------|-------------------------------|-------------------------------|----------|
| 一.3 文案可读 · 标题≤15字+含利益点 | 「尊享 V10 专属权益服务升级体验」(13字但抽象) | 「立省 ¥187」(6字+数字+利益) | 下单页省钱卡主标 |
| 一.4 交互 · 一屏主按钮=1 | 升房卡同时出现「预约」「取消」「咨询」3个同权重按钮 | 态1b 主按钮=「立即预约」胶囊indigo-1；其他态次按钮降级为indigo-4 | 订详升房卡 |
| 三.用户心理 · 差异感知替代损失厌恶 | 「最后 1 间!不订明天涨价!」 | 「已为你提交升房申请 · 酒店 24h 内告知」 | 升房态2 |
| 五.P1 模糊状态 | 「升房处理中」无时间 | 「酒店 24h 内告知结果 · 提交于 5.10 09:12」 | 升房态2 |
| 五.P0 暗黑模式 | 态2「撤销申请」链接文字+下划线(弱化) | 态2「撤销申请」保持胶囊按钮尺寸,仅色阶降级 | 升房态2 |
```

**抽取规则**：
- 必须覆盖 DP 一.1~一.4 四维度各至少 1 条，每条配反例+正例 Demo
- 必须显式回应 DP 三.用户心理的⚠️重要决策（若 DP 中有）
- 必须列出 DP 五.反模式中本项目可能踩中的 ≥2 条，每条配 Demo

##### 字典 C：DNA token → **Mini 视觉 Demo**

不只列 token 名称，必须给**实际渲染后的 mini 片段**（ASCII 或 HTML snippet），让用户看到色/字号/间距真实效果。

##### 字典冲突 → **双方案 Demo 对照 + 用户选一**

三类字典之间若存在冲突，必须在 6.0 末尾以「冲突表 + 双方案 Demo」显式暴露给用户裁决，**禁止擅自抹平**。格式：


##### 6.0 末尾 · 用户决策确认清单（必出）

字典 + 冲突表整理完后,**必须**汇总成一个「决策清单」,每条配 2-3 个 Demo 选项让用户选,不允许 6.1 之前留悬而未决的问题。格式：

```
🚦 请基于以下 Demo 拍板（不拍板不得进 6.1）：

决策点 1：[问题]
  ○ 选项 A [Demo 截片]
  ○ 选项 B [Demo 截片]

```

**字典下游使用**：
- 6.2 草图每个元素必须能指到三类字典中的至少 A+B 两项
- 6.4 自检追溯表字段扩展为「PRD/DRD 行号 | DP 条款 | DNA 组件」

#### 6.1 触点清单确认

从 DRD「六、设计规格」中提取所有触点，生成确认清单交给用户：

```
📋 Step 5.5 分层表中 L3 及以上条目共 [N] 条，请确认需要生成方案的范围：

| # | 触点名称 | **层级** | **产物格式** | 对应 Design_DNA | 基底文件 | 是否生成 |
|---|----------|----------|----------|-----------------|----------|----------|
| 1 | [触点名] | L3 | 组件 diff | Design_DNA_Order | hotel-order.html 中的 X 区域 | ❓待确认 |
| 2 | [触点名] | L5 | 整页 HTML | Design_DNA_Booking | hotel-booking.html | ❓待确认 |
| 3 | [触点名] | L3 | 组件 diff | 无现有 DNA | 需新建片段 | ❓待确认 |

- [触点X]：[原因]
```

**规则**：
- **层级硬闸**：Step 5.5 标注为 L0/L1/L2 的条目禁止进入本 Step，若发现软性匹配为 L0-L2 的触点被错拉入，直接退回 Step 5.5 重分层
- 每个触点必须明确绑定到一个 Design_DNA 项目（Order/Booking/其他），无法绑定时标注并询问用户
- DRD 中标注「长期」的触点默认不生成，除非用户明确要求
- 如果某个触点的改动仅涉及文案/标签（L1），禁止在此层默认升级为组件/整页 HTML
- **L3 条目的产物**：只生成组件片段 HTML（纯 DIV，不含 <html>/<body>/viewport）+ 插入位置文字说明，禁止搬整页 DNA 底图
- **L5/L6 条目的产物**：完整页面 HTML + 状态枚举

#### 6.2 草图方案（两形式 + 三项自检 · 禁止低保真 HTML）

用户确认范围后，**先输出草图 + 自检**，不直接生成高保真 HTML。

**草图两形式组合**（不允许纯文字描述）：

| 形式 | 角色 |
|------|------|
| **C · Mermaid** | 全局链路视图（必出 · 跨触点跳转/状态流转） |
| **A · ASCII 线框** | 单页结构（必出 · 每个触点 × 每个状态） |

> ⚠️ **禁止低保真 HTML demo**：低保真既不准(无真 DNA token)又浪费迭代(先做灰阶再推翻做高保真)。Mermaid + ASCII 足够表达结构与层级，确认后**直接跳 6.3 高保真 DNA token 版本**，所有视觉/间距/色/组件细节都在高保真里一次到位。

**推进顺序**：C（全局链路） → A（每触点/每状态 ASCII） → 三项自检 → 用户拍板 → 6.3 高保真 HTML。

每个触点的草图必须同步输出两项自检：

##### 自检 1：决策闭环 5 要素（任何决策型界面适用）

用户完成一个决策需要回答 5 个问题，缺任一者 = 设计不完整：

```
| 要素 | 回答的问题 | 本触点如何处理 | PRD/DRD 出处 |
|------|----------|----------------|---------------|
| Who · 身份 | 我是谁/我凭什么有资格 | [元素描述] | PRD L[行号] |
| What · 动作 | 能做什么 | [元素描述] | PRD L[行号] |
| Cost/Value · 代价 | 付出/得到多少 | [元素描述] | PRD L[行号] |
| When/Where · 边界 | 什么条件下有效 | [元素描述] | PRD L[行号] |
| Action · 行动 | 下一步怎么办 | [元素描述] | PRD L[行号] |
```

**规则**：5 要素缺一即视为不完整，必须补齐或回 PRD 确认是否存在。

##### 自检 2：极简反模式 3 红线（极简设计时强制执行）

「极简」是手段不是目的。以下 3 条不可突破：

```
| 红线 | 定义 | 反例（不允许） | 判定的问题 |
|------|------|--------------|-------------|
| R1 · 否定平权 | 肯定操作必须配可发现的否定/退出操作 | 只保留 1 个主按钮，取消靠下滑/点遮罩 | 删除后用户仍能安全退出吗？ |
| R2 · 决策闭环完整 | 5 要素不可删（见自检 1） | 为求简洁删掉时效/金额/身份 | 删除后用户仍能独立完成决策吗？ |
| R3 · 人格化保留 | 称谓/情感语言属于产品本身，不是冗余 | 把「尊贵的 X 会员」简化为「X 会员」 | 删除后用户仍感到被尊重吗？ |
```

**极简合格公式**：`删除后用户仍能完整决策 + 仍感到被尊重 = 极简合格`。

##### 派发格式（每个触点必出）

```
📝 触点 [N]：[触点名称]
基底文件：Design_DNA_[X]/[文件名].html

▸ 草图（C Mermaid 全局链路 + A ASCII 每态线框）

▸ 改动概述：
  - [组件A]：[具体改动 + PRD/DRD 出处行号]

▸ 状态覆盖：
  - 状态[状态名]：[展示内容]

▸ 自检 1 · 决策闭环 5 要素表（如上）

▸ 自检 2 · 极简反模式 3 红线核对：
  - R1 否定平权：✅/❌（说明）
  - R2 决策闭环：✅/❌（说明）
  - R3 人格化：✅/❌（说明）

▸ 自检 3 · DesignPrinciple 四维度预评（草图阶段即预判，<7 不得进 6.3）：
  - 链路连贯性 [0-10]：[评分理由 + 绑定 DP 一.1 哪条]
  - 信息表达完整性 [0-10]：[评分理由 + 绑定 DP 一.2 哪条]
  - 文案可读性 [0-10]：[评分理由 + 绑定 DP 一.3 哪条 + 字数核对]
  - 交互好用性 [0-10]：[评分理由 + 绑定 DP 一.4 哪条 + 主按钮数核对]
  - 反模式核对：[列出本触点可能踩中的 DP 五.反模式条目 + 规避方案]

▸ 待确认问题：
  - ❓ [不确定的设计细节]
```

**规则**：
- 每个触点的草图必须覆盖 DRD 中定义的所有状态枚举
- 所有新增元素必须标注插入位置 + PRD/DRD 出处行号
- 任何一条红线未通过 = 不得进入 6.3，必须先修正草图
- 三项自检不是"事后检查"，是「草图输出时即集成」
- **自检 3 任一维度 <7 即为不合格**，必须说明原因 + 修正方案后重新评分，否则不得进 6.3

#### 6.3 HTML 生成

1. **复制或新建基底文件**：

## 执行规则

1. **不跳步**：必须按 Step 1→2→3→3.5→4→5→6 顺序执行，每步输出后等待用户确认再继续
   - Step 3 必须输出 PRD 审核表→ Step 3.5 让设计师选择→ Step 4 只输出被选中的方案
   - Step 6 内部强制分步：6.0 关键词字典 → 6.1 触点确认 → 6.2 草图三形式+双项自检 → 等用户确认 → 6.3 HTML 生成 → 6.4 完整性+追溯表自检
2. **可中断**：用户可以在任何步骤插入补充信息或纠正理解
3. **引用原则**：每个建议必须关联到 DesignPrinciple.md 中的具体原则
4. **不做美工**：输出信息架构和交互规格，不输出具体视觉方案（颜色/字号/间距像素值等）
5. **模块化**：如果用户只需要某个步骤（如只做设计稿审阅），可以跳过前置步骤
6. **可执行**：Step 5 的输出必须达到「另一个智能体读完就能画出设计稿」的标准

## 输出质量标准

### DRD 完整性检查清单

输出前必须自检以下条件全部满足：

- [ ] 每个决策点有明确推荐方向和理由
- [ ] 每个涉及的页面/触点都有独立的设计规格
- [ ] 每个新增元素有：位置、结构、内容规则、约束条件
- [ ] 每个元素有完整的状态枚举（默认/空/异常/加载）
- [ ] 文案有推荐和避免的对比表
- [ ] 全局约束条件明确（数量限制、字数限制、信息密度）
- [ ] 全链路信息一致性已标注
- [ ] 优先级排序有明确依据

### 质量红线

- 每条建议必须**具体、可执行**（不接受"优化用户体验"这种空话）
- 必须说明**为什么**（关联用户痛点或设计原则）
- 风险提示必须有**规避方案**
- 设计规格必须达到**另一个智能体读完能直接输出设计稿**的颗粒度

## 自进化机制

每次 Pipeline 执行完成并输出 DRD 后，执行以下自进化流程：

### 触发条件
用户对输出结果给出反馈（修改、补充、质疑、纠正）时触发。

### 执行步骤

1. **差异识别**：对比用户反馈与当前输出，提炼出 SKILL 本身缺失的信息类型：
   - 新的行业经验/业务规则？
   - 新的设计原则/反模式？
   - 流程步骤的缺失/顺序问题？
   - 输出模板结构不够完整？
   - 量化标准需要调整？

2. **提炼建议**：将差异归纳为具体的改进条目，格式如下：
   ```
   📝 本次执行发现以下可沉淀到 SKILL 的经验：
   
   1. [类型：原则/流程/模板/标准] — [具体内容]
      写入位置：[SKILL.md 的哪个章节 / DesignPrinciple.md 的哪个章节]
   
   2. ...
   
   是否将以上内容写入 SKILL 进行迭代？(Y/N/选择性写入)
   ```

3. **用户确认后写入**：
   - Y → 全部写入对应文件
   - N → 不写入，仅本次生效
   - 选择性 → 用户指定写入哪几条

4. **写入目标**：
   - 行业经验/业务规则/项目决策 → `BussinessExp.md` 对应章节
   - 设计原则/反模式/评分标准 → `DesignPrinciple.md` 对应章节
   - 流程调整/模板扩展/执行规则 → 本文件 `SKILL.md` 对应章节
   - 项目特有规则 → `Step1_放入资料/[项目名]/项目规则.md`（新建）

### 不写入的情况
- 纯粹个人偏好（非通用经验）
- 与现有原则矛盾且用户未明确表态替代
- 一次性的临时决策
